2026-07-06

  • 今日看了什么：提前带读 repos/TradingAgents/LEARNING_PLAN_zh.md 的“项目心智模型”和“推荐阅读顺序”，并用 main.py、trading_graph.py、setup.py、dataflows/interface.py、llm_clients/factory.py 做代码锚点。
  • 我真正理解了什么：TradingAgents 把系统拆成 graph、agents、dataflows、llm_clients，不是为了显得复杂，而是把流程编排、角色职责、事实数据、模型接入四件事隔离开。一次运行从 TradingAgentsGraph.propagate() 进入图流程，图负责状态流转，Agent 负责生成报告和观点，dataflows 负责拿事实，llm_clients 负责屏蔽模型供应商差异。
  • 这个设计对复盘 Agent 的启发：个人复盘 Agent 第一版也应拆成“流程编排、复盘角色、事实来源、模型适配/输出约束”四层。即使不用 LangGraph，也要避免把读取操作记录、审计逻辑、行情事实、最终报告全部塞进一个 prompt。
  • 还没懂什么：LangGraph 里 AgentState 的字段如何被各节点增量更新；工具调用循环和辩论循环的条件路由还需要读 trading_graph.py、setup.py、conditional_logic.py。
  • 下次建议看：repos/TradingAgents/main.py 和 repos/TradingAgents/tradingagents/graph/trading_graph.py，只追踪 TradingAgentsGraph.__init__() 到 propagate() 的主线。

2026-07-07

  • 今日看了什么：解释 TradingAgentsGraph.__init__() 中创建 ConditionalLogic(max_debate_rounds, max_risk_discuss_rounds) 的代码，以及 conditional_logic.py 里的路由函数。
  • 我真正理解了什么：ConditionalLogic 是图流程的“路由裁判”。它不执行 Agent，也不调用 LLM，只根据当前 AgentState 判断下一步走工具节点、清理消息节点、下一个辩论角色，还是进入最终经理节点。
  • 这个设计对复盘 Agent 的启发：复盘 Agent 也可以把“什么时候继续追问事实、什么时候进入反方审计、什么时候生成最终报告”写成独立路由逻辑，而不是散落在 prompt 里。
  • 还没懂什么：investment_debate_state["count"] 和 risk_debate_state["count"] 具体在哪些 Agent 节点里递增。
  • 下次建议看：repos/TradingAgents/tradingagents/graph/setup.py 中 workflow.add_conditional_edges(...) 如何使用这些路由函数。

Propagator 小段

  • 今日看了什么：repos/TradingAgents/tradingagents/graph/propagation.py 和 TradingAgentsGraph._run_graph() 中 create_initial_state()、get_graph_args() 的调用。
  • 我真正理解了什么：Propagator 负责把一次运行启动起来：先创建完整的初始 AgentState，再提供 LangGraph 执行参数，比如 recursion_limit 和 callbacks。真正执行图的是 self.graph.invoke(...) 或 debug 模式下的 self.graph.stream(...)。
  • 这个设计对复盘 Agent 的启发：复盘 Agent 也需要一个“运行启动器”，统一初始化输入记录、日期、历史行为模式、审计状态和输出占位，避免每个节点自己补默认值。
  • 还没懂什么：debug 模式下 stream_mode="values" 输出的 chunk 为什么需要手动合并。
  • 下次建议看：AgentState 字段如何被 analyst、researcher、risk、portfolio manager 节点逐步写入。

AgentState 小段

  • 今日看了什么：agents/utils/agent_states.py 的 AgentState、InvestDebateState、RiskDebateState，以及 market analyst、bull researcher、trader、portfolio manager 对 state 的典型读写。
  • 我真正理解了什么：AgentState 是整张 LangGraph 的共享工作台和数据契约。每个节点从里面读自己需要的上下文，只返回自己新增或更新的字段，LangGraph 再把这些增量合并回总状态。
  • 这个设计对复盘 Agent 的启发：复盘 Agent 应该也有一个明确的 ReviewState，保存原始操作记录、结构化交易事件、事实检查结果、正反审计意见、最终复盘报告和行为模式更新。
  • 还没懂什么：MessagesState 对 messages 字段的追加合并规则，以及普通字段覆盖合并的细节。
  • 下次建议看：setup.py 如何用 StateGraph(AgentState) 声明全图共享状态。

Reflector / SignalProcessor 小段

  • 今日看了什么：graph/reflection.py、graph/signal_processing.py，以及 TradingAgentsGraph._resolve_pending_entries()、process_signal() 的调用点。
  • 我真正理解了什么：Reflector 用 LLM 对已经知道结果的历史决策做 2-4 句反思，写回 memory log；SignalProcessor 不再调用 LLM，只用 parse_rating() 从 Portfolio Manager 的最终决策文本里提取 Buy / Overweight / Hold / Underweight / Sell。
  • 这个设计对复盘 Agent 的启发：复盘系统也应区分“事后反思模块”和“结构化信号提取模块”。前者可以用 LLM 归纳教训，后者应尽量确定性解析，避免每天输出字段漂移。
  • 还没懂什么：memory log 的 pending entry 什么时候能拿到足够行情结果并被反思更新。
  • 下次建议看：agents/utils/memory.py 如何把 pending 决策、结果收益和 reflection 串成长期记忆。

setup_graph 小段

  • 今日看了什么：GraphSetup.setup_graph() 如何用 StateGraph(AgentState) 注册 analyst、tool、message clear、researcher、trader、risk、portfolio manager 节点，并用普通边和条件边连接成完整工作流。
  • 我真正理解了什么：LangGraph 里节点是“接收 state、返回 state 增量”的函数；边决定节点执行顺序；条件边会调用一个路由函数，根据当前 state 返回下一节点。TradingAgents 的 analyst 阶段是 Analyst -> ToolNode -> Analyst 的工具循环，结束后先清理 messages，再进入下一个 analyst。
  • 这个设计对复盘 Agent 的启发：复盘 Agent 可以先做一张小图：读取记录 → 清洗 → 审计 → 反方审计 → 生成报告 → 更新行为模式库；以后再给事实检查节点增加条件循环。
  • 还没懂什么：LangGraph workflow.compile() 后内部如何保存节点和边，以及 graph.invoke() 如何逐步合并 state。
  • 下次建议看：选一个 analyst 节点，例如 market_analyst.py，跟踪它如何触发 tool call 并把 market_report 写回 AgentState。

conditional edge / clear msg 小段

  • 今日看了什么：getattr(self.conditional_logic, f"should_continue_{spec.key}") 如何根据 analyst key 动态选择 should_continue_market/social/news/fundamentals，以及 create_msg_delete() 为什么清理 messages。
  • 我真正理解了什么：getattr 只是在建图时拿到对应路由函数；真正判断发生在运行时，路由函数读取 state["messages"][-1].tool_calls，有工具调用就去对应 ToolNode，没有工具调用就去对应 Msg Clear 节点。
  • 这个设计对复盘 Agent 的启发：每个阶段可以有自己的“继续查资料还是进入下一阶段”的判断函数；阶段完成后应清理临时对话，只把稳定结果写入结构化 state。
  • 还没懂什么：RemoveMessage 在 LangGraph 的 message reducer 中具体如何删除旧消息。
  • 下次建议看：market_analyst.py 中 result.tool_calls 在什么情况下为空，什么时候会触发 tools_market。
