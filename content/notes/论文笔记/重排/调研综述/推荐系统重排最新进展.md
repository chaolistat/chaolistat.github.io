- 结论一句话：重排已经从“精排后面补规则”逐步变成一个独立的列表级优化层，核心矛盾是 **候选排列空间巨大、目标多且动态、线上延迟极紧、离线目标难以代表真实用户满意度**。

## 2. Vault 内可接续阅读
- 可控多目标重排：[[CMR]] 是静态线性权重无法适应线上变化的代表解法，用 policy hypernetwork 支持给定偏好权重下的动态重排。
- 重排入口：[[AgenticRecTune]] 把 pre-ranking、ranking、re-ranking 的系统级配置交给 agent 和线上 A/B 闭环优化，适合理解“模型外层参数怎么调”。
- 精排 / LTR 背景：[[EMER]]、[[Pantheon]]、[[UnifiedRL]] 属于精排侧的多目标融合和 LTR 路线，能补足重排常面对的目标冲突、在线收益对齐和候选比较背景。
- 精排 / 生成式搜索背景：[[OneSearch]]、[[OneSearch-V2]] 虽然是电商搜索/生成式检索，但它们强调的“多阶段流水线局部最优”和“reward system 对齐”与重排问题相通。

## 3. 业界一般做法

工业推荐通常仍然是多阶段流水线：

1. 召回从百万/十亿级物料中取回千到万级候选。
2. 预排用轻模型快速过滤。
3. 精排用重模型给每个候选独立或弱交互打分。
4. 重排在几十到几百个候选上生成最终曝光列表。

重排层做的事情不是简单把精排分数再排序一次，而是补上精排阶段难以建模的列表级因素：

- **候选间交互**：相邻 item 是否重复、是否互相替代、是否形成疲劳。
- **多目标融合**：点击、时长、转化、分享、关注、复访、商家/创作者生态等目标经常冲突。
- **多样性和新颖性**：避免同质内容堆叠，也给长尾和探索留位置。
- **业务约束**：广告、活动、冷启动保护、品类占比、供给侧公平、用户负反馈过滤。
- **布局约束**：信息流、双列瀑布流、货架、搜索结果页、短视频上下滑的展示结构不同，重排目标也不同。
- **在线稳定性**：延迟、缓存、AB 分桶、guardrail 指标和回滚能力通常比离线 NDCG 更重要。

最常见的生产形态是混合式：

- 先用精排分数形成基础相关性。
- 再用规则、penalty、MMR/DPP、线性标量化或可学习融合器处理多样性和业务约束。
- 对候选量较小、收益较高的场景，引入 list-wise 模型或 generator-evaluator。
- 最后一定依赖线上 A/B，而不是只看离线排序指标。

[[CMR]] 的摘要明确指出，多阶段 ranking pipeline 已是现代推荐系统常用策略，final stage 要平衡用户偏好、多样性、新颖性等要求；线性标量化是业界常用的多目标融合方法，但静态权重无法快速适应环境变化。这个判断基本可以作为工业重排的基线设定。

## 4. 方法谱系

### 4.1 规则和贪心重排

这是最稳的工程基线。典型做法是从精排候选里逐个挑 item，每一步最大化：

$$
score(i) = relevance(i) - \lambda \cdot redundancy(i, S) + business\_bonus(i) - penalty(i)
$$

其中 $S$ 是已经选中的列表。优点是解释性强、延迟低、可控；缺点是很难学习复杂的 item-item 依赖，也很容易把线上效果调成一堆难维护的规则。

### 4.2 可控多目标重排

这一类把重排看成“不同目标权重下的可控排序器”。[[CMR]] 用 policy hypernetwork 根据偏好权重生成重排模型参数，避免每次业务权重变化都重新训练。[[MultiTRON]] 用 Pareto front approximation，让一个模型通过额外输入的 preference vector 覆盖不同 CTR/CVR 权衡点。

这条路线很适合工业系统，因为线上经常不是追单一目标，而是今天要保 GMV，明天要保留存，活动期间要保供给侧曝光。关键问题是权重如何被产品目标、用户分群和时间周期驱动，而不是变成另一套人工调参。

### 4.3 神经列表级重排

这条路线用 Transformer、set encoder 或 request-wise 模型同时看一组候选，让每个 item 的最终分数依赖同组其它候选。它比 point-wise 精排更贴近最终曝光列表，但也更受候选规模、推理延迟和样本构造影响。

现有 vault 里的 [[EMER]] 就是这一方向的强相关例子：它不是只学单个 item 的绝对好坏，而是把同一 request 的候选打包，用候选维度 self-attention 做比较学习。

### 4.4 生成式重排

生成式重排把最终列表看成一个序列生成任务。优点是天然能建模位置依赖和 item 间依赖；缺点是排列空间巨大，AR 解码容易慢且有误差传播，NAR 解码快但会削弱 slot 之间依赖。

近期工业论文的主线几乎都在解决这个矛盾：

- [[NAR4Rec]]：把 autoregressive generator 改成 non-autoregressive，提高实时部署可行性，并在快手 3 亿 DAU 短视频 App 全量部署。
- [[GReF]]：保留 AR 的序列建模能力，但用 ordered multi-token prediction 降低推理成本，并用 Rerank-DPO 把序列级偏好接进训练。
- [[CONGRATS]]：指出高似然序列可能重复且低质量，用 graph-structured decoder 扩大多路径解码空间，缓解 “likelihood trap”。
- [[UniRank]]：把 AR 和 NAR 统一到 confidence-ordered denoising 框架里，按最有把握的位置逐步填充列表。

### 4.5 LLM 辅助重排

LLM 进入重排主要有三种方式：

1. 直接作为 zero-shot / prompted reranker，用自然语言要求模型兼顾相关性、多样性、公平等目标。
2. 作为语义增强器，补传统协同过滤或 GNN 难以理解的 item 语义、互补关系、长尾属性。
3. 作为配置和实验 agent，调重排层的权重、阈值、业务规则和 A/B 实验方案。

当前判断要保守：LLM 在高价值、低候选量、强语义场景有潜力，例如互补商品、搜索结果重排、解释性推荐；但在主信息流里直接让大模型排 top-100，通常仍会被延迟、成本、稳定性和候选顺序敏感性卡住。[[InvariRank]] 这类工作正是在解决 LLM listwise reranking 的候选输入顺序敏感问题。

## 5. 2024-2026 前沿趋势

### 5.1 从 point-wise 分数走向 slate-level 最优化

精排模型输出的通常是每个 item 的局部分数，但最终用户看到的是一个列表。最新论文普遍把重排表述为 request-specific candidate pool 到 ordered slate 的映射，目标是最大化列表整体满意度，而不是单个 item 的点击概率。

### 5.2 从静态权重走向动态可控

线性加权仍是工程默认方案，但前沿方向是在不重新训练的情况下动态调整偏好：按用户分群、场景、节假日、供需状态或业务策略输入不同 preference vector。[[CMR]] 和 [[MultiTRON]] 是这条线的代表。

### 5.3 从两阶段 generator-evaluator 走向端到端训练

早期生成式重排常用 generator 生成多个候选列表，再由 evaluator 选最优。问题是两者分离会造成训练和推理不一致。[[GReF]] 用 Rerank-DPO 把 evaluator 的偏好信号接进 generator 训练，[[CONGRATS]] 也强调 differentiable cascade/evaluator 以直接从用户偏好学习。

### 5.4 从 AR/NAR 二分走向统一生成框架

AR 依赖强但慢，NAR 快但容易 slot independence。[[UniRank]] 的价值在于把二者视为同一 denoising 框架的不同推理特例：不是固定从左到右排，而是每一步填最有把握的位置。

### 5.5 多样性不再只是后处理指标

[[CONGRATS]] 的 “likelihood trap” 很关键：最大似然训练倾向于高频、重复、看似安全的序列，但人感知质量可能更差。未来重排会更重视显式 diversity、novelty、coverage、fairness，以及这些目标和用户满意度之间的因果关系。

### 5.6 LLM 重排的核心问题变成可靠性和效率

[[LLM4Rerank]]、[[LLM-Diversity-Re-ranking|LLM diversity reranking]] 和 [[LLM-Enhanced-Reranking-for-Complementary-Product-Recommendation|complementary product reranking]] 都说明 LLM 能提供语义理解和目标可编程性。但 2026 的 [[InvariRank]] 把问题推进了一步：LLM listwise 排序可能因为候选输入顺序变化而输出不同结果，这会影响线上可信度。未来更可能是“小模型/蒸馏模型在线跑，大模型离线生成训练信号或做高价值场景兜底”。

### 5.7 重排和在线实验系统越来越耦合

重排层直接决定曝光，因此离线指标很容易失真。[[AgenticRecTune]] 这类工作提示一个方向：未来重排优化不只是模型结构问题，也包括自动生成实验配置、分析 A/B 指标、更新策略库和控制风险。

## 8. 对后续研究的判断

如果目标是做工业可落地的重排，优先关注三件事：

- **延迟和候选规模**：方法如果不能在几十毫秒内处理几十到几百个候选，只适合离线或高价值小流量场景。
- **目标对齐**：离线 NDCG、Recall、CTR proxy 不够，最好设计能连接在线收益、用户满意度或长期目标的训练/评估方案。
- **可控性**：业务目标会变，重排模型需要能被 preference vector、policy、规则边界或 agent 配置稳定控制。

如果目标是写论文，比较有潜力的切口：

- 把生成式重排和多目标 Pareto 控制结合起来，而不是只优化单一排序指标。
- 用大模型生成偏好数据或解释信号，再蒸馏到小型 list-wise reranker。
- 研究重排的 offline-online consistency，尤其是曝光偏差、反事实评估和线上 guardrail。
- 针对瀑布流、短视频、feed、货架等不同展示结构设计 layout-aware reranking。
- 把用户疲劳、内容生态、供给侧公平纳入统一 slate-level objective。
